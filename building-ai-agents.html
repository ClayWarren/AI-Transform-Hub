What Are AI Agents?

AI agents are large language models (LLMs) that dynamically interact with their environment through tools, operating in a loop to accomplish complex tasks. Unlike rigid workflows with predefined paths, agents use LLMs to make decisions, select tools, and adapt based on environmental feedback.

Key Components

Environment

The system the agent operates in (e.g., a file system for coding or a customer support platform).

Tools

Interfaces that allow the agent to interact with the environment (e.g., reading files, executing commands, or querying databases).

System Prompt

Defines the agent's goals, constraints, and ideal behavior to guide its operation within the environment.

Basic Agent Loop Structure

env = Environment() tools = Tools(env) system_prompt = "Goals, constraints, and how to act" while True: action = llm.run(system_prompt + env.state) env.state = tools.run(action)

Performance Scaling

As LLMs improve, agents leveraging them become more capable, particularly for high-volume, low-to-medium complexity tasks where they can achieve superhuman performance.

When to Build an Agent

Before building an agent, evaluate the task using a checklist to ensure it warrants an agentic approach:

Agent Evaluation Checklist

  • Is the task complex enough? Agents are ideal for tasks requiring multiple steps, dynamic decision-making, or tool usage that a single LLM call cannot handle efficiently.
  • Is the task valuable enough? The task's value should justify the development effort and operational costs (e.g., latency, token usage).
  • Are all parts of the task doable? Ensure the necessary tools, data, and permissions are available to complete the task.
  • What is the cost of error? Assess whether errors are tolerable or require strict guardrails and sandboxed testing to mitigate risks.

If simpler solutions (e.g., optimized single prompts with retrieval or in-context examples) suffice, avoid unnecessary complexity. Agents are best suited for open-ended tasks like coding (e.g., editing multiple files) or customer support (e.g., resolving tickets with external data), where flexibility and action-oriented workflows are critical.

Core Principles for Building Agents

1. Simplicity

Start with minimal complexity, using direct LLM API calls to maintain transparency and ease debugging. Avoid frameworks unless they demonstrably improve outcomes, as they can obscure prompts and responses.

2. Transparency

Ensure the agent's planning and tool-use steps are explicit, making it easier to debug and build trust in the system.

3. Tool Design (ACI)

Craft clear, well-documented tools with descriptive names, parameters, and usage examples to minimize errors and guide the LLM effectively.

Building Blocks and Patterns

Augmented LLM

The foundation of an agent is an LLM enhanced with augmentations:

  • Retrieval: Access to external data (e.g., knowledge bases, file contents)
  • Tools: Functions for interacting with the environment (e.g., read_file, edit_file)
  • Memory: Retaining conversation history or task context to maintain continuity

Workflow Patterns

For structured tasks, consider these workflows before building a full agent:

Prompt Chaining

Decompose tasks into sequential LLM calls, with optional programmatic checks (gates) to ensure accuracy (e.g., generate marketing copy, then translate it).

Routing

Classify inputs and direct them to specialized prompts or tools (e.g., routing customer queries to refund or technical support processes).

Parallelization

Sectioning: Break tasks into independent subtasks run in parallel. Voting: Run the same task multiple times to aggregate diverse outputs.

Orchestrator-Workers

A central LLM delegates tasks to worker LLMs for complex, unpredictable tasks (e.g., editing multiple files in a coding project).

Evaluator-Optimizer

One LLM generates responses, while another evaluates and refines them iteratively (e.g., refining translations or search results).

Autonomous Agents

For open-ended tasks, build autonomous agents that:

  • Plan and execute tasks independently, using tools based on environmental feedback
  • Pause for human input at checkpoints or when encountering blockers
  • Include stopping conditions (e.g., maximum iterations) to prevent runaway processes

Testing Requirements

Test agents extensively in sandboxed environments to manage costs and errors. Agents excel in scenarios where the number of steps is unpredictable, and the LLM must make decisions dynamically.

Practical Implementation: A Code-Editing Agent

Thorsten Ball's guide demonstrates building a code-editing agent in ~300 lines of Go code using Anthropic's Claude 3.7 Sonnet. The implementation is straightforward yet powerful, showcasing the simplicity of agent design.

Setup

Initialize a Go project with Anthropic's SDK:

mkdir code-editing-agent cd code-editing-agent go mod init agent touch main.go

Create an Agent struct with an LLM client (anthropic.Client) and tool definitions, reading user input from the terminal.

Core Loop

The agent operates in a loop:

  1. Read user input (e.g., "Create fizzbuzz.js")
  2. Append input to the conversation history
  3. Send the conversation and tool definitions to the LLM
  4. Execute tools if requested, append results to the conversation, and repeat
func (a *Agent) Run(ctx context.Context) error { conversation := []anthropic.MessageParam{} fmt.Println("Chat with Claude (use 'ctrl-c' to quit)") readUserInput := true for { if readUserInput { fmt.Print("\\u001b[94mYou\\u001b[0m: ") userInput, ok := a.getUserMessage() if !ok { break } userMessage := anthropic.NewUserMessage(anthropic.NewTextBlock(userInput)) conversation = append(conversation, userMessage) } message, err := a.runInference(ctx, conversation) if err != nil { return err } conversation = append(conversation, message.ToParam()) toolResults := []anthropic.ContentBlockParamUnion{} for _, content := range message.Content { switch content.Type { case "text": fmt.Printf("\\u001b[93mClaude\\u001b[0m: %s\\n", content.Text) case "tool_use": result := a.executeTool(content.ID, content.Name, content.Input) toolResults = append(toolResults, result) } } if len(toolResults) == 0 { readUserInput = true continue } readUserInput = false conversation = append(conversation, anthropic.NewUserMessage(toolResults...)) } return nil }

Tools

The agent uses three essential tools:

read_file

Description: Reads the contents of a file at a given relative path.

Input Schema: { "path": string }

Function: Reads the file and returns its contents as a string.

list_files

Description: Lists files and directories at a given path (defaults to current directory).

Input Schema: { "path": string, optional }

Function: Returns a JSON array of file and directory names (directories with trailing /).

edit_file

Description: Replaces old_str with new_str in a file or creates a new file if it doesn't exist.

Input Schema: { "path": string, "old_str": string, "new_str": string }

Function: Reads the file, replaces the text, and writes back; creates a new file if old_str is empty.

Example Interaction

Command: "Create fizzbuzz.js with a FizzBuzz implementation."

  • Agent checks for existing files with list_files({})
  • Creates fizzbuzz.js with edit_file({"path":"fizzbuzz.js","old_str":"","new_str":"..."}), containing a JavaScript implementation
  • Output: A file that runs with node fizzbuzz.js, printing numbers 1 to 100 with "Fizz", "Buzz", or "FizzBuzz" for multiples of 3, 5, or both

The agent dynamically combines tools without explicit instructions, relying on Claude's reasoning to decide when to list, read, or edit files.

Prompt Design for Agents

Parahelp's prompt design strategies emphasize structured, explicit prompts to guide agents, particularly for customer support tasks with clear success metrics (e.g., % of tickets resolved).

Planning Prompt

  • Structure: Use XML with <plan>, <step>, and <if_block> tags to outline tool calls and conditional paths
  • Guidelines: Specify only the goal of the next steps, not the overall task. Follow company policies and use valid tool names
  • Challenges: Handling large context (~1.5K tokens) with dynamic ticket information and planning all possible paths based on tool outcomes
<plan> <step> <action_name>search_helpcenter</action_name> <description>Search helpcenter for information about feature_name and how to resolve error_name</description> </step> <if_block condition='<helpcenter_result> found'> <step> <action_name>reply</action_name> <description>Reply to the user with instructions from <helpcenter_result></description> </step> </if_block> <if_block condition='no <helpcenter_result> found'> <step> <action_name>reply</action_name> <description>Reply to the user that we need more information and ask for {{troubleshooting_info_name_from_policy_2}}</description> </step> </if_block> </plan>

Manager Prompt

The LLM acts as a manager to approve or reject tool calls based on policy and context:

  • Analyze ticket context and internal messages
  • Check tool calls against policies and a checklist
  • Return <manager_verify>accept</manager_verify> or <manager_verify>reject</manager_verify>

Best Practices

  • Use markdown and XML for structured, parsable prompts
  • Specify the model's thinking order (e.g., analyze context, check policy, decide)
  • Emphasize critical instructions with "ALWAYS" or "IMPORTANT"
  • Test prompts extensively to identify edge cases

Tool Design (Agent-Computer Interface)

Tools are critical for agent performance. Anthropic's guidelines emphasize designing tools as carefully as prompts:

Clarity

Use descriptive names, parameters, and examples. Include usage scenarios, edge cases, and boundaries from other tools.

Simplicity

Choose formats that are easy for the LLM to generate (e.g., string replacement over diffs for edit_file).

Natural Fit

Align tool output with natural text formats seen in training data (e.g., markdown over JSON for code).

Error Prevention

Design tools to minimize mistakes (e.g., require absolute file paths to avoid relative path errors).

Testing Requirement

Run tools with varied inputs to identify failure modes and iterate on descriptions. For a coding agent, edit_file using string replacement (old_str, new_str) is simpler and more reliable than generating diffs.

Proven Use Cases

Customer Support

Why It Works: Combines conversation with actions like querying knowledge bases, issuing refunds, or updating tickets. Success is measurable via ticket resolution rates.

Example: An agent searches a help center for error resolutions, replies with instructions, or escalates if no solution is found, using tools like search_helpcenter and reply.

Coding

Why It Works: Verifiable through automated tests, with clear problem spaces and feedback loops. Agents can iterate on solutions using test results.

Example: An agent creates and edits fizzbuzz.js or congrats.js, combining list_files, read_file, and edit_file to solve tasks like implementing FizzBuzz or decoding ROT13 strings.

Key Takeaways

Essential Guidelines

  • Start Simple: Use direct LLM calls and add agentic complexity only when simpler solutions fall short
  • Design Clear Tools and Prompts: Craft well-documented tools and structured prompts (e.g., XML plans) to guide LLM behavior and improve reliability
  • Test Extensively: Use sandboxed environments and clear success metrics to iterate on prompts and tools
  • Leverage Workflows and Agents: Combine patterns like prompt chaining, routing, or autonomous agents to fit your use case
  • Focus on ACI: Invest as much effort in tool design as in prompt engineering to ensure the LLM uses tools effectively

By following these principles, developers can build effective, reliable AI agents tailored to specific tasks, leveraging the power of LLMs while maintaining control, transparency, and robustness.
